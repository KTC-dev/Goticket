# Supabase Client Library Usage Analysis

## Backend Usage
- **Package**: `@supabase/supabase-js` (version ^2.0.0) in backend/package.json
- **Initialization**: 
  - Created in `backend/server.js` using `createClient(supabaseUrl, supabaseKey)`
  - Uses `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` from environment variables
  - Made available to routes via middleware: `req.supabase = supabase`
- **Usage**: 
  - Extensively used in route handlers (`backend/routes/events.js`, `backend/routes/tickets.js`, etc.)
  - Used for all database operations: SELECT, INSERT, UPDATE, DELETE
  - Used in seed scripts (`backend/seeds/eventSeed.js`)
  - Used in utility scripts for inspection/testing

## Frontend Usage
- **Package**: `@supabase/supabase-js` (version ^2.106.2) in frontend/package.json
- **Initialization**:
  - Created in `frontend/src/supabase.js` using `createClient(supabaseUrl, supabaseAnonKey)`
  - Uses `REACT_APP_SUPABASE_URL` and `REACT_APP_SUPABASE_ANON_KEY` from environment variables
  - Exports the supabase instance for use throughout the application
- **Usage**:
  - Used in `frontend/src/AuthContext.js` for all authentication operations:
    - Session management (`supabase.auth.getSession()`)
    - Auth state change listener (`supabase.auth.onAuthStateChange()`)
    - Sign in/up/out (`supabase.auth.signInWithPassword()`, `supabase.auth.signUp()`, `supabase.auth.signOut()`)
    - Password reset (`supabase.auth.resetPasswordForEmail()`)
    - User updates (`supabase.auth.updateUser()`)
  - Used in `frontend/src/pages/Events.js` for ticket purchasing functionality:
    - API calls to backend endpoints for fetching events and creating tickets
    - Dynamic API base URL configuration using environment variables
    - Payment information state management after ticket reservation
    - UI updates for payment confirmation via WhatsApp/email

## Key Observations
1. The Supabase client is the primary method for database interactions in the backend
2. The Supabase client is used for authentication in the frontend (via AuthContext)
3. Both endpoints use the official Supabase JavaScript client library
4. The backend uses the service role key for elevated privileges
5. The frontend uses the anon key for client-side operations
6. The system follows Supabase best practices for separating concerns between frontend and backend access
7. The Events.js frontend component demonstrates integration with Supabase-powered backend APIs for ticket purchasing workflow
